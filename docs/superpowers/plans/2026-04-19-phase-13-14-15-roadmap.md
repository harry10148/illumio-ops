# Phase 13–15 Roadmap — PCE Cache + SIEM Forwarder + Consumer Migration

**As of:** 2026-04-19
**Driver:** Periodic API pulls for SIEM forwarding should double as a shared local cache for reports and alerts, so all three consumers share one PCE rate budget (500 req/min).

---

## Dependency graph

```
Phase 13 (infra) ─┬─► Phase 14 (reports read cache)
                  └─► Phase 15 (alerts read cache)
```

Phase 14 and 15 are **independent** of each other once Phase 13 ships — they can run in parallel or deferred indefinitely.

Phase 13 **must** ship in a state where existing reports/alerts keep working unchanged (direct-API path preserved). Cache is additive.

## Phase 13 — Infrastructure (new codebase territory)

**Plan:** `docs/superpowers/plans/2026-04-19-phase-13-pce-cache-and-siem.md`

**Goal:** Ship the shared PCE cache (ingestor + 6-table SQLite schema + retention + rate limiter) **and** the SIEM forwarder (2 formatters × 4 transports + dispatcher + DLQ + CLI/GUI) as a single coherent PR. SIEM is the first consumer of the cache, which validates the cache contract end-to-end.

**What explicitly does NOT change in Phase 13:**
- `src/report/` — still calls `ApiClient` directly. Migration is Phase 14.
- `src/analyzer.py` — still calls `ApiClient` directly for events/traffic. Migration is Phase 15.
- `src/events/poller.py` — existing monitor-loop event poll is independent of cache in Phase 13 (dual-read is fine; cache is additive).

**Target tag:** `v3.11.0-siem-cache`

## Phase 14 — Reports read from cache

**Plan:** `docs/superpowers/plans/2026-04-19-phase-14-reports-on-cache.md`

**Goal:** `AuditGenerator` (events) and `ReportGenerator` (traffic) prefer cache over direct API. When the requested time range falls outside cache retention, the GUI/CLI surfaces a "backfill required" affordance rather than silently skipping or silently bursting the API.

**Target tag:** `v3.12.0-reports-cache`

---

## Phase 15 — Alerts subscribe to cache

**Plan:** `docs/superpowers/plans/2026-04-19-phase-15-alerts-on-cache.md`

**Goal:** `Analyzer.run_analysis()` consumes new events/flows from cache via `ingested_at > last_check` polling, eliminating redundant API calls that previously duplicated the cache ingestor's work. Near-real-time alerting (30s polling floor) becomes viable without breaching rate limits.

**Target tag:** `v3.13.0-alerts-cache`

## Rollout / risk notes

1. **Schema migration strategy**: SQLite + SQLAlchemy 2.x `Base.metadata.create_all()` is enough for Phase 13 (table created if missing, no-op if present). Any destructive column change after v3.11 ships requires an explicit migration script under `scripts/migrations/`.
2. **Rate-limit regression risk**: the global rate limiter must gate ingestor calls but also the existing `ApiClient` caller sites — otherwise concurrent monitor + ingestor could still exceed 500/min. Phase 13 Task 3 wires the limiter into `api_client._request()` on a feature flag so the monitor path keeps working unchanged if the flag is off.
3. **Traffic volume spike**: first production deployment must start with `pce_cache.traffic_poll_interval_seconds=3600` (1-hour batches) and `traffic_filter.actions=["blocked","potentially_blocked"]` to bound the blast radius. A follow-up tuning doc will ship in Phase 13 `docs/PCE_Cache.md`.
4. **DLQ bloat**: cap at 10k rows per destination by default; oldest rows auto-purge when cap hit and a loguru ERROR fires. Administrator must manually drain DLQ via CLI `illumio-ops siem dlq replay|purge`.
5. **Disk pressure**: 7 days of blocked-only traffic at 10k flows/day ≈ 70k rows ≈ < 50 MB. 90 days of agg ≈ same order. Document `df` expectations in `docs/PCE_Cache.md`.
